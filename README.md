Hola! El objetivo de este README.md es explicar (muy) brevemente mi entrega, y las decisiones que tomé en cuanto a diseño. 

La decisión de mayor peso que tomé fue reestructurar la base de datos, de manera tal que cada Brand (empresa) trae consigo un conjunto de pares (pregunta, respuesta). Así, el código queda realmente limpio, pero el drawback más evidente es que se está incrementando la carga sobre el backend (que por cierto, ya tiene bastantes dificultades con tener un archivo de texto como base de datos!). En mi opinión, este es un buen trade-off porque la rapidez no es tremendamente importante, y así se tiene una estructura muy, muy simple y fácil de trabajar y escalar. Además, si se utilizara otro método, como por ejemplo, obtener "options" del backend, y mostrar dinámicamente el contenido en el front según ellas, ocurre que (i) se complica innecesariamente el código, y (ii) hay una dificultad respecto a cómo mantener el lenguaje natural en las FAQs. 

Por otro lado, sin perjuicio lo anterior, considero fuertemente que tener un backend para una aplicación de este tipo no es del todo necesario. Claramente, se entiende que debía utilizarse un backend para ilustrar mis competencias como programador, por lo que decidí dejarlo, pero francamente considero que una solución mucho más cómoda, rápida de desarrollar, y escalable, hubiera sido no tener un backend, sino que escribir todo en Next, a modo de quedar con una aplicación de tipo estática. El mayor beneficio que le veo a tener un backend sería poder agregar, eliminar o editar brands de manera programática, pero, francamente, creo que en la vida real tales cambios serían muy poco frecuentes (o, por lo menos, no lo suficientemente frecuentes como para justificar el costo que significa tener un backend). Una solución mucho más simple y escalable era no tener backend.

Finalmente, otra posible optimización pequeña que pude haber hecho consiste en lo siguiente: actualmente, al obtener todas las brands (para mostrarlas en la vista principal), también se están fetcheando los pares (pregunta, respuesta), a pesar de que **no se muestran en esa vista**. Posteriormente, al seleccionar una brand, nuevamente se fetchean los pares (pregunta, respuesta). Una optimización evidente aquí sería fetchear sólo lo necesario en la vista principal y, después, al clickear en una brand, obtener las FAQs. Siendo franco, no lo hice por tiempo.
